# /note-run スキル

note.com向けの記事を**完全自動で**生成するスキル。トピック選定から最終稿まで人間の介入不要。

---

## Phase 0: コンテキスト読み込み

以下を読み込んで現状を把握する：
1. `context/profile.md` — プロフィール・ポジショニング
2. `context/existing-articles.md` — 過去記事（テーマ重複回避）
3. `context/voice-samples.md` — 文体サンプル
4. `context/strategy.md` — 収益化戦略・ノウハウ
5. `context/performance.md` — 過去記事のパフォーマンスデータ（PV・スキ数・売上）
6. Notion「note記事管理」DB（data_source: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）
   - 過去記事のパフォーマンス（PV・スキ数・売上）
   - 現在進行中の記事があるか
   - 直近のトレンド（どんなテーマが反応良かったか）
7. Notion「ネタ帳」DB（data_source: `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595`）
   - ステータスが「未使用」のネタ候補を取得
   - 過去に見送ったネタの理由も確認（同じネタを再スキャンしないため）

## Phase 1: AIニューストレンドスキャン

WebSearchで以下を検索し、直近24〜48時間のAI関連ニュースを収集する：

**検索クエリ:**
- "Claude" OR "Anthropic" 新機能 リリース
- "Claude Code" update OR 新機能 OR アップデート
- "OpenAI" OR "ChatGPT" 新機能
- "Gemini" OR "Google AI" アップデート
- "生成AI" 最新ニュース
- "AI agent" OR "AIエージェント" 最新

**収集する情報:**
- ニュースのタイトル・URL・日付
- 何が新しいのか（1行要約）
- 速報性（リリースから何時間経っているか）

### ネタ帳への記録
スキャンで見つけた全候補（採用する1本だけでなく全部）をNotionネタ帳DBに記録する:
- mcp__notion__notion-create-pages で各候補を登録:
  - parent: { "data_source_id": "1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595" }
  - ネタ名: ニュースタイトル
  - ソースURL: 元記事URL
  - 発見日: 今日の日付
  - 記事タイプ候補: 速報/実録/ノウハウ
  - メモ: 1行要約・差別化の可能性
  - ステータス: "未使用"（Phase 2で採用されたものだけ"採用済み"に更新）

## Phase 2: トピック選定（自動判定）

Phase 1の結果と `context/strategy.md` のノウハウを使って、最適なトピックを自動選定する。

### まずネタ帳をチェック
Phase 1で新規スキャンする前に、ネタ帳DBの「未使用」ネタを確認する。
- mcp__notion__notion-fetch で `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595` を参照
- 未使用のネタの中に、今日出すべきもの（緊急度が上がったもの等）がないか確認
- ある場合は新規スキャン結果と合わせてスコアリング
- ない場合はPhase 1のスキャン結果のみでスコアリング

### 判定基準（スコアリング）

各候補に以下の5項目で1〜5点をつけ、合計スコアが最も高いものを選ぶ：

| 項目 | 判定基準 | 根拠 |
|---|---|---|
| **市場の大きさ** | そのトピックに関心がある人の推定数。X検索やnote検索で関連投稿数を確認 | bookの3要素 |
| **悩みの深さ** | 「お金を払ってでも解決/知りたい」レベルか。HARM法則（Money/Ambition）に当てはまるか | bookのHARM法則 |
| **緊急度** | 「今すぐ知りたい」か。新機能リリース直後は緊急度が高い | bookの緊急度 |
| **差別化可能性** | 既にnoteで同テーマの記事が溢れていないか。自分の実体験で語れるか | 競合調査 |
| **過去実績との相性** | 過去記事で反応が良かったテーマとの類似性。`context/performance.md`を参照 | PDCAフィードバック |

**スコア20点以上** → 有料記事（500円）として作成
**スコア15〜19点** → 無料記事として作成
**スコア14点以下** → 見送り。次の候補に進む

### 記事タイプの自動判定
- 新機能リリースから48時間以内 → **速報記事（sokuho）**
- 過去の実務で使った体験がある → **実録記事（jituroku）**
- 汎用的なハウツー → **ノウハウ記事（knowhow）**

### Notion記録: 記事レコード作成
トピックが決まったら、Notionに新しいレコードを作成する:
- mcp__notion__notion-create-pages で以下を登録:
  - parent: { "data_source_id": "812aa728-8d3e-42e4-a9cd-6a91c303b2c2" }
  - 記事名: （仮タイトル）
  - ステータス: "進行中"
  - 記事タイプ: "速報" / "実録" / "ノウハウ"
  - スコア: （算出したスコア）
  - トピック: （選定理由・元ネタ情報）
  - 価格: （推奨価格）
- 作成されたページURLを記録しておく（Phase 8で更新するため）

### ネタ帳のステータス更新
- 採用したネタ → ステータスを "採用済み" に更新（mcp__notion__notion-update-page）
- スコアが低くて見送ったネタ → ステータスを "見送り" に更新、見送り理由を記入
- それ以外 → "未使用" のまま残す（将来使う可能性あり）

## Phase 3: 競合調査

選定したトピックについて、note上の競合を調査する。

**やること:**
1. WebSearchで `site:note.com {トピックのキーワード}` を検索
2. 上位5記事のタイトル・スキ数・価格を確認
3. 差別化ポイントを特定：
   - 競合が書いていない角度は何か
   - 自分の実体験で追加できる情報は何か
   - 「最強の使い方」を実演できるか

**売上推測（bookの公式）:**
- スキの数 × 記事の価格 × 1.5 ≒ おおよその売上
- スキ100以上の有料記事があれば「売れるテーマ」と判断

## Phase 4: 構成設計

以下のテンプレートから記事タイプに合った構成を選び、トピックに合わせてカスタマイズする。

### 速報記事の構成（800〜1,500字）
1. **結論ファースト** — 何が出たか、何が変わるか（3行以内）
2. **公式情報のまとめ** — リリースノートから
3. **実際に使ってみた** — 一次情報（「こうやったらこうなった」）
4. **最強の使い方** — 具体的なユースケースを3つ提案
5. **まとめ** — 一言で価値を要約 + 次回予告

### 実録記事の構成（2,000〜4,000字）
1. **フック** — 読者の共感を得る導入
2. **背景** — 何をしたかったか
3. **やったこと** — Claude Codeでどう解決したか
4. **結果** — Before/After（数字で示す）
5. **つまずいたこと** — 失敗談
6. **学び** — この経験から得た知見
7. **次回予告**

### ノウハウ記事の構成（2,500〜5,000字）
1. **フック** — 読者が抱えている問題
2. **この記事で得られること**
3. **ステップ1〜N** — 具体的な手順
4. **つまずきポイント**
5. **応用** — 他にどう使えるか
6. **まとめ** + 次回予告

## Phase 5: 執筆

### 絶対ルール
- 文体は `context/voice-samples.md` に合わせる
- 一人称: 「自分」「僕」
- AI定型フレーズ禁止: 「〜と言えるでしょう」「〜ではないでしょうか」「重要なのは」の連発
- 括弧（）での補足説明の多用禁止
- 数字・実績は正確に（盛らない）
- クライアント情報は匿名化
- 有料記事の場合: 無料エリア（冒頭30〜40%）で十分な価値を見せてから有料ライン

### 出力フォーマット: noteエディタ対応HTML

記事はnoteエディタにHTMLペーストで入力するため、以下のHTML形式で出力すること。

**使用するHTMLタグ:**
- `<h2>` — メインセクションの見出し
- `<h3>` — サブセクション（具体例、手順の各ステップ等）
- `<p>` — 通常の段落
- `<b>` — 強調（キーポイント、Before/After等）
- `<blockquote>` — 引用（公式情報の引用、ユーザーへの指示文、印象的な一文）
- `<ul><li>` — 箇条書き（できること一覧、特徴等）
- `<ol><li>` — 番号付きリスト（手順、「この先で手に入るもの」等）
- `<pre><code>` — コードブロック（コマンド、設定値、コード例）
- `<hr>` — 区切り線（セクション間、有料ラインの前後）
- `<p style="text-align: center;">` — 中央寄せ（有料ラインの境界メッセージ）
- `<br>` — 段落内改行（Before/Afterの並列表示等）

**装飾の使い分けガイド:**
- h2は記事全体で3〜6個。大きなセクション区切り
- h3はh2の下に2〜4個。具体例や手順の各ステップ
- 引用（blockquote）は「ユーザーがAIに送る指示文」「公式情報の要約」「印象的な一言」に使う
- コードブロック（pre>code）は「コマンド」「設定値」「コード例」に使う
- 中央寄せは有料ラインの境界メッセージにのみ使う（多用禁止）
- 区切り線（hr）は有料ラインの前後、大きなセクション転換時に使う
- 番号付きリストは「この先で手に入るもの」や手順に使う

**有料記事の構成:**
- 有料ライン前に `<hr>` + 中央寄せメッセージ + 「この先で手に入るもの」リスト + `<hr>`

### 有料記事の無料エリア設計（bookのノウハウ）
- 無料エリアだけでも「役に立った」と思わせる
- 有料エリアには「実際のコード/プロンプト全文」「具体的な数字」「テンプレート」等の実践的価値を入れる
- 無料→有料の境界に「この先で手に入るもの」リストを入れる

## Phase 6: 7スウィープ編集

以下の7つの観点で順番にチェック・修正する：

1. **明確さ**: 曖昧な表現を具体的に
2. **声**: AI感のある表現を自然な大学生の言葉に
3. **So What?**: 各段落に「だから何？」の答えがあるか
4. **根拠**: 数字・事例・ソースがあるか
5. **具体性**: 抽象→具体
6. **感情**: 読者の感情を動かすポイントがあるか
7. **リスク**: 誤情報・誇張・情報漏洩がないか

- 見出しが内容を正確に要約しているか

## Phase 8: 保存・出力

### 保存形式

2つのファイルを保存する：

1. `drafts/{記事タイプ}/draft_{日付}_{トピック要約}.md` — メタ情報＋Markdown（バックアップ・Git管理用）
2. `drafts/{記事タイプ}/draft_{日付}_{トピック要約}.html` — noteエディタ貼り付け用HTML

HTMLファイルはメタ情報なし、`<h2>`〜`</p>`のみの純粋なHTMLボディ。
`<html>`や`<body>`タグは不要。noteエディタにそのままペーストできる形式。

Markdownファイルのフォーマット（冒頭メタ情報）:
```
---
title: 記事タイトル
type: sokuho/jituroku/knowhow
price: 0 or 500
tags: [タグ1, タグ2, ...]
created: YYYY-MM-DD
status: final
score: {Phase 2のスコア}
---
```

### コンソール出力

記事完成後、以下を表示する：
- 記事タイトル
- 記事タイプ・推奨価格
- 文字数
- Markdownファイルパス
- HTMLファイルパス ← noteに貼り付けるファイル
- 推奨タグ一覧
- 「HTMLファイルの内容をnoteエディタにペーストして投稿してください」のメッセージ

### Notion記録: レコード更新
記事完成後、Phase 2で作成したNotionレコードを更新する:
- mcp__notion__notion-update-page で以下を更新:
  - 記事名: （最終タイトル）
  - ステータス: "完了" ← 投稿待ちの意味
  - メモ: "最終稿完成。noteにコピペして投稿してください。ファイル: {ファイルパス}"

## Phase 9: PDCAフィードバック記録

`context/performance.md` に以下を追記するよう促す:
- 記事タイトル・URL・公開日
- 投稿後1週間のPV・スキ数
- 売上（有料記事の場合）
- 学び（何が良かった/悪かったか）

このデータは次回のPhase 2（トピック選定）で自動参照される。

### Notion記録: パフォーマンス更新の案内
記事投稿後、以下をNotionに記録するよう促す:
- noteURL: 投稿後のURL
- 公開日: 投稿日
- 投稿1週間後に以下を更新:
  - PV: ページビュー数
  - スキ数: スキの数
  - 売上: スキ数 × 価格 × 1.5
  - メモ: 学び・次回への示唆

### PDCAメモの記録
記事管理DBのメモ欄に、以下のフォーマットでPDCA記録を残すよう促す:

**投稿直後に記録:**
- 投稿日時
- noteURL
- 使ったタグ一覧

**1週間後に記録:**
- PV数
- スキ数
- コメント数
- 高評価数（購入手続きバナー下に表示）
- 売上推計（スキ数 × 価格 × 1.5）

**振り返り:**
- 何が良かったか（タイトル？内容？タイミング？）
- 何が悪かったか
- 次回への改善アクション（具体的に1つ）

このPDCAデータは次回の /note-run 実行時にPhase 0で自動参照される。
