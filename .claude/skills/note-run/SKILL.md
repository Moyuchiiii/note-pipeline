# /note-run スキル

## オプション: 深掘りリサーチモード

`deep_research: true` を指定すると、Phase 3（競合調査）とPhase 6（品質ゲート・根拠スウィープ）が強化版になる。

- デフォルト: true（常に深掘りリサーチを実行）
- オフにしていい場面: 速報記事で時間優先の場合、事実関係が明確でソースが公式のみの場合

note.com向けの記事を**完全自動で**生成するスキル。トピック選定から最終稿まで人間の介入不要。

## Phase 0: コンテキスト読み込み + PDCA自動学習

以下を読み込んで現状を把握する：
1. `context/profile.md` — プロフィール・ポジショニング
2. `context/existing-articles.md` — 過去記事（テーマ重複回避）
3. `context/voice-samples.md` — 文体サンプル
4. `context/strategy.md` — 収益化戦略・ノウハウ
5. `context/performance.md` — 過去記事のパフォーマンスデータ（PV・スキ数・売上）
6. Notion「note記事管理」DB（data_source: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）
   - 過去記事のパフォーマンス（PV・スキ数・売上）
   - 現在進行中の記事があるか
   - 直近のトレンド（どんなテーマが反応良かったか）
7. Notion「ネタ帳」DB（data_source: `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595`）
   - ステータスが「未使用」のネタ候補を取得
   - 過去に見送ったネタの理由も確認（同じネタを再スキャンしないため）

### PDCA自動学習（過去データから成功パターンを抽出）

Notion記事管理DBの全レコードを取得し、以下を自動分析する：

**パフォーマンス分析:**
- スキ数が最も多かった記事のテーマ・構成・タイトルパターン
- PVが最も多かった記事の特徴
- 有料記事の売上ランキング
- 無料記事のうちスキ数が多かったもの（有料化候補）

**成功パターンの抽出:**
- タイトルの型: 「〇〇してみた」「〇〇の最強の使い方」等、どの型がスキ数多いか
- 記事の長さ: 何文字の記事が最も反応良いか
- 記事タイプ: 速報/実録/ノウハウのどれが反応良いか
- タグ: どのタグの記事が反応良いか

**直近の出稿状況:**
- 直近5本の記事タイプの内訳（速報○本/実録○本/ノウハウ○本）
- 直近5本の無料/有料比率
- 最後に投稿した日からの経過日数

これらの分析結果を Phase 2 のスコアリングに反映する。

- "Claude" 新機能 OR リリース OR アップデート
- "Anthropic" リリース OR 発表 OR announcement

**収集する情報:**
- ニュースのタイトル・URL・日付

## Phase 2: トピック選定（自動判定）

Phase 1の結果と `context/strategy.md` のノウハウを使って、最適なトピックを自動選定する。

### まずネタ帳をチェック
Phase 1で新規スキャンする前に、ネタ帳DBの「未使用」ネタを確認する。
- mcp__notion__notion-fetch で `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595` を参照
- 未使用のネタの中に、今日出すべきもの（緊急度が上がったもの等）がないか確認
- ある場合は新規スキャン結果と合わせてスコアリング
- ない場合はPhase 1のスキャン結果のみでスコアリング

### 判定基準（スコアリング）

各候補に以下の5項目で1〜5点をつけ、合計スコアが最も高いものを選ぶ：

| 項目 | 判定基準 | 根拠 |
|---|---|---|
| **市場の大きさ** | そのトピックに関心がある人の推定数。X検索やnote検索で関連投稿数を確認 | bookの3要素 |
| **悩みの深さ** | 「お金を払ってでも解決/知りたい」レベルか。HARM法則（Money/Ambition）に当てはまるか | bookのHARM法則 |
| **緊急度** | 「今すぐ知りたい」か。新機能リリース直後は緊急度が高い | bookの緊急度 |
| **差別化可能性** | 既にnoteで同テーマの記事が溢れていないか。自分の実体験で語れるか | 競合調査 |
| **過去実績との相性** | 過去記事で反応が良かったテーマとの類似性。`context/performance.md`を参照 | PDCAフィードバック |

### スコアに基づく判定 + バランス制御

**基本判定:**
- スコア20点以上 → 有料記事（500円）として作成
- スコア15〜19点 → 無料記事として作成
- スコア14点以下 → 見送り。次の候補に進む

**バランス制御（基本判定を上書きする場合あり）:**
Phase 0で分析した直近の出稿状況に基づき、以下のルールで無料/有料を調整する：

1. **有料連投防止**: 直近5本のうち有料が2本以上 → 今回はスコアに関係なく無料にする
2. **無料連投時の有料チャンス**: 直近5本が全部無料 → スコア18点以上なら有料にする（閾値を2点下げる）
3. **初期フォロワー制限**: Notionの記事管理DBの総記事数が10本未満 → 有料率を20%以下に制限（5本に1本まで）
4. **記事タイプ分散**: 直近5本が同じ記事タイプ → 別タイプを優先（速報ばかりにならないように）

**PDCAフィードバック加算:**
Phase 0で抽出した成功パターンに該当する場合、スコアにボーナスを加算：
- 過去最もスキ数が多かったテーマと同系統 → +2点
- 過去最もスキ数が多かったタイトルパターンと同じ型 → +1点
- 過去反応が悪かったテーマと同系統 → -2点

### 記事タイプの自動判定
- Claude/Anthropicの新機能リリースから48時間以内 → **速報記事（sokuho）**
- Claude Codeを実務で使った体験がある → **実録記事（jituroku）**
- Claude/Claude Codeの使い方・活用法 → **ノウハウ記事（knowhow）**

### エラーハンドリング

**ニュースが見つからない場合:**
Phase 1でスキャン結果が0件の場合：
1. ネタ帳DBの「未使用」ネタを確認
2. 未使用ネタがあれば、その中からスコアリングして最適なものを選ぶ
3. 未使用ネタもない場合 → 「今日は速報ネタがありません。ノウハウ記事を書きますか？」とユーザーに確認

**全候補がスコア14点以下の場合:**
→ 「本日は投稿を見送ります。理由: {各候補のスコアと見送り理由}」と報告して終了

**スコアリング結果の表示:**
採否に関わらず、全候補のスコア内訳を表示する：
```
📊 トピック候補スコアリング結果:
1. [トピック名] — 合計: 22点 (市場5/悩み4/緊急5/差別4/実績4) → ✅ 採用（有料500円）
2. [トピック名] — 合計: 16点 (市場3/悩み3/緊急4/差別3/実績3) → 📋 ストック
3. [トピック名] — 合計: 12点 (市場2/悩み2/緊急3/差別3/実績2) → ❌ 見送り
```

### Notion記録: 記事レコード作成
トピックが決まったら、Notionに新しいレコードを作成する:
- mcp__notion__notion-create-pages で以下を登録:
  - parent: { "data_source_id": "812aa728-8d3e-42e4-a9cd-6a91c303b2c2" }
  - 記事名: （仮タイトル）
  - ステータス: "進行中"
  - 記事タイプ: "速報" / "実録" / "ノウハウ"
  - スコア: （算出したスコア）
  - トピック: （選定理由・元ネタ情報）
  - 価格: （推奨価格）
- 作成されたページURLを記録しておく（Phase 8で更新するため）

### ネタ帳のステータス更新
- 採用したネタ → ステータスを "採用済み" に更新（mcp__notion__notion-update-page）
- スコアが低くて見送ったネタ → ステータスを "見送り" に更新、見送り理由を記入
- それ以外 → "未使用" のまま残す（将来使う可能性あり）

## Phase 3: 競合調査

選定したトピックについて、note上の競合を調査する。

**やること:**
1. WebSearchで `site:note.com {トピックのキーワード}` を検索
2. 上位5記事のタイトル・スキ数・価格を確認
3. 差別化ポイントを特定：
   - 競合が書いていない角度は何か
   - 自分の実体験で追加できる情報は何か
   - 「最強の使い方」を実演できるか

**売上推測（bookの公式）:**
- スキの数 × 記事の価格 × 1.5 ≒ おおよその売上
- スキ100以上の有料記事があれば「売れるテーマ」と判断

### deep_research: true の場合（強化版競合調査）

上記の基本調査に加えて、以下を実行する：

1. `research/03-search-strategy.md` に従い、トピックについて多角的な検索クエリを生成する
   - 言い換え・具体化・拡張の3軸で各2〜3クエリ
   - 日本語メイン + 英語補助
   - 適応的クエリ拡張（ヒット不足時の自動リトライ）
2. `research/04-data-collection.md` に従い、信頼度タグ付きで情報を収集する
   - ✅ 一次データ / 🔶 二次データ / ⚠️ 参考情報
   - ソースURL必須、省略禁止
   - 収集結果を `drafts/research/research_{日付}_{トピック}.md` に保存
3. `research/05-verification.md` に従い、クロス検証を実行する
   - 2ソース以上で確認できた情報に [検証済] タグ
   - 悪魔の代言人テスト（反証を意図的に検索）
   - 検証結果を収集結果ファイルに追記

収集・検証した情報はPhase 5（執筆）とPhase 6（品質ゲート・根拠スウィープ）で参照する。

## Phase 4: 構成設計 + タイトル最適化

### タイトルA/Bテスト

記事のタイトル候補を**3本**生成し、以下の基準でスコアリングして最適なものを選ぶ。

**タイトル生成の型（過去の成功パターンから選択）:**
- 数字型: 「〇〇したら毎日2時間浮いた話」
- 方法型: 「〇〇の最強の使い方」
- 体験型: 「大学生が〇〇してみたら」
- 疑問型: 「〇〇って何ができるの？」
- 速報型: 「〇〇が出たので即日試してみた」
- 比較型: 「〇〇 vs △△、実際に使って比較」

**タイトルスコアリング（各20点満点）:**

| 項目 | 判定基準 |
|---|---|
| **検索キーワード** | ターゲットが検索しそうなワードが含まれているか（0-5点） |
| **クリック欲求** | 「読みたい」と思わせる好奇心・ベネフィットがあるか（0-5点） |
| **具体性** | 数字・固有名詞が入っているか（0-5点） |
| **過去成功パターン類似度** | Phase 0で抽出した成功タイトルの型に合致するか（0-5点） |

**表示フォーマット:**
```
🏷️ タイトル候補:
1. [タイトルA] — 16/20点 ✅ 採用
2. [タイトルB] — 13/20点
3. [タイトルC] — 11/20点
```

### 構成テンプレート

以下のテンプレートから記事タイプに合った構成を選び、トピックに合わせてカスタマイズする。

### 速報記事の構成（800〜1,500字）
1. **結論ファースト** — 何が出たか、何が変わるか（3行以内）
2. **公式情報のまとめ** — リリースノートから
3. **実際に使ってみた** — 一次情報（「こうやったらこうなった」）
4. **最強の使い方** — 具体的なユースケースを3つ提案
5. **まとめ** — 一言で価値を要約 + 次回予告

### 実録記事の構成（2,000〜4,000字）
1. **フック** — 読者の共感を得る導入
2. **背景** — 何をしたかったか
3. **やったこと** — Claude Codeでどう解決したか
4. **結果** — Before/After（数字で示す）
5. **つまずいたこと** — 失敗談
6. **学び** — この経験から得た知見
7. **次回予告**

### ノウハウ記事の構成（2,500〜5,000字）
1. **フック** — 読者が抱えている問題
2. **この記事で得られること**
3. **ステップ1〜N** — 具体的な手順
4. **つまずきポイント**
5. **応用** — 他にどう使えるか
6. **まとめ** + 次回予告

## Phase 5: 執筆

### 絶対ルール
- 文体は `context/voice-samples.md` に合わせる
- 一人称: 「自分」「僕」
- AI定型フレーズ禁止: 「〜と言えるでしょう」「〜ではないでしょうか」「重要なのは」の連発
- 括弧（）での補足説明の多用禁止
- 数字・実績は正確に（盛らない）
- クライアント情報は匿名化
- 有料記事の場合: 無料エリア（冒頭30〜40%）で十分な価値を見せてから有料ライン

### 出力フォーマット: noteエディタ対応HTML

記事はnoteエディタにHTMLペーストで入力するため、以下のHTML形式で出力すること。

**使用するHTMLタグ:**
- `<h2>` — メインセクションの見出し
- `<h3>` — サブセクション（具体例、手順の各ステップ等）
- `<p>` — 通常の段落
- `<b>` — 強調（キーポイント、Before/After等）
- `<blockquote>` — 引用（公式情報の引用、ユーザーへの指示文、印象的な一文）**＋ 箇条書き的な列挙にも使う**（noteの引用ブロックは視認性が高いため、ポイント列挙・できること一覧・特徴リスト等は `<blockquote>` で1項目ずつ囲む）
- `<ul><li>` — **原則使わない**。箇条書き的な列挙は `<blockquote>` を使うこと。`<ul><li>` は「あわせて読みたい」のリンクリスト等、装飾不要の短いリストにのみ使用
- `<ol><li>` — 番号付きリスト（手順、「この先で手に入るもの」等）。ステップ形式で順序が重要な場合のみ使用
- `<pre><code>` — コードブロック（コマンド、設定値、コード例）
- `<hr>` — 区切り線（セクション間、有料ラインの前後）
- `<p style="text-align: center;">` — 中央寄せ（有料ラインの境界メッセージ）
- `<br>` — 段落内改行（Before/Afterの並列表示等）

**装飾の使い分けガイド:**
- h2は記事全体で3〜6個。大きなセクション区切り
- h3はh2の下に2〜4個。具体例や手順の各ステップ
- 引用（blockquote）は「ユーザーがAIに送る指示文」「公式情報の要約」「印象的な一言」に加え、**箇条書き的な列挙（ポイント・特徴・できること一覧等）にも積極的に使う**。noteでは引用ブロックが視覚的に映えるため、`<ul><li>`より`<blockquote>`を優先する
- コードブロック（pre>code）は「コマンド」「設定値」「コード例」に使う
- 中央寄せは有料ラインの境界メッセージにのみ使う（多用禁止）
- 区切り線（hr）は有料ラインの前後、大きなセクション転換時に使う
- 番号付きリストは「この先で手に入るもの」や手順に使う

**有料記事の構成:**
- 有料ライン前に `<hr>` + 中央寄せメッセージ + 「この先で手に入るもの」リスト + `<hr>`

### 有料記事の無料エリア設計（bookのノウハウ）
- 無料エリアだけでも「役に立った」と思わせる
- 有料エリアには「実際のコード/プロンプト全文」「具体的な数字」「テンプレート」等の実践的価値を入れる
- 無料→有料の境界に「この先で手に入るもの」リストを入れる

## Phase 6: 品質ゲート（100点ルーブリック + 7スウィープ編集）

### Step 1: 7スウィープ編集

以下の7つの観点で順番にチェック・修正する：

1. **明確さ**: 曖昧な表現を具体的に。「いろいろ」「さまざまな」→具体名を入れる
2. **声**: AI感のある表現を自然な大学生の言葉に。voice-samples.mdと比較して不自然な箇所を修正
3. **So What?**: 各段落に「だから何？」の答えがあるか。読者が「で？」と思う段落は削除or加筆
4. **根拠**: 主張に数字・事例・ソースがあるか。「すごい」「便利」→具体的な時間節約や金額で示す
   - deep_research: trueの場合、`drafts/research/` の検証済み事実リストを参照し、記事内の主張と照合する。[検証済]タグの情報を優先的に使い、[未検証]の情報には記事内で「〜との報道がある」等の留保表現を使う
5. **具体性**: 抽象→具体。「効率化できる」→「40分が5分になった」
6. **感情**: 読者の感情を動かすポイントが最低2箇所あるか。驚き・共感・発見のいずれか
7. **リスク**: 誤情報・誇張・情報漏洩がないか。クライアント名・金額の匿名化を再確認

### Step 2: 100点ルーブリック採点

以下の5カテゴリで採点し、合計60点未満なら自動rewrite（最大2回まで）。

**コンテンツ品質（30点満点）:**
- 導入で読者の関心を引けているか（0-6点）
- 具体例・数字が十分に含まれているか（0-6点）
- 結論が明確で行動可能か（0-6点）
- 冗長な部分がないか（0-6点）
- 次回予告・CTA（フォロー誘導）があるか（0-6点）

**SEO（25点満点）:**
- タイトルに検索キーワードが含まれているか（0-10点）
- 見出しが内容を正確に要約しているか（0-5点）
- 冒頭100文字で記事の価値が伝わるか（0-5点）
- タグが適切か（0-5点）

**読者体験（20点満点）:**
- 読みやすい段落長か（1段落3〜4文以内）（0-5点）
- 装飾（見出し・引用・リスト）で視覚的にメリハリがあるか（0-5点）
- Before/Afterや数字で具体的な変化が伝わるか（0-5点）
- 有料記事の場合、無料エリアだけでも価値があるか（0-5点）

**voice一貫性（15点満点）:**
- voice-samples.mdのトーンと一致しているか（0-5点）
- AI定型フレーズが含まれていないか（0-5点）
- 一人称が「自分」「僕」で統一されているか（0-5点）

**リスク（10点満点）:**
- 誤情報・未検証の主張がないか（0-5点）
- クライアント情報・機密情報の匿名化（0-5点）

**判定:**
- 80点以上: 高品質。そのまま公開可
- 60〜79点: 合格。軽微な修正で公開可
- 60点未満: 不合格。自動rewriteを実行（問題箇所を特定して修正し、再採点）
- 2回rewriteしても60点未満: ユーザーに報告して判断を仰ぐ

**採点結果の表示:**
```
📊 品質スコア: 78/100点 ✅ 合格
  コンテンツ品質: 24/30
  SEO: 20/25
  読者体験: 16/20
  voice一貫性: 12/15
  リスク: 6/10 ← 「要確認: ○○の記述」
```

## Phase 8: 保存・出力

### 保存形式

2つのファイルを保存する：

1. `drafts/{記事タイプ}/draft_{日付}_{トピック要約}.md` — メタ情報＋Markdown（バックアップ・Git管理用）
2. `drafts/{記事タイプ}/draft_{日付}_{トピック要約}.html` — noteエディタ貼り付け用HTML

HTMLファイルはメタ情報なし、`<h2>`〜`</p>`のみの純粋なHTMLボディ。
`<html>`や`<body>`タグは不要。noteエディタにそのままペーストできる形式。

Markdownファイルのフォーマット（冒頭メタ情報）:
```
---
title: 記事タイトル
type: sokuho/jituroku/knowhow
price: 0 or 500
tags: [タグ1, タグ2, ...]
created: YYYY-MM-DD
status: final
score: {Phase 2のスコア}
---
```

### コンソール出力

記事完成後、以下を表示する：
- 記事タイトル
- 記事タイプ・推奨価格
- 文字数
- Markdownファイルパス
- HTMLファイルパス
- 推奨タグ一覧（noteのハッシュタグ用）
- 有料ラインの位置（何番目のブロックの後か）

### 投稿タイミング提案

Phase 0で取得した過去記事の投稿日時とスキ数の相関を分析し、最適な投稿タイミングを提案する。

**分析ロジック:**
1. 過去記事の「公開日の曜日」と「スキ数」の相関を確認
2. データが5本未満の場合はデフォルト推奨値を使用

**デフォルト推奨値（データ不足時）:**
- 速報記事: リリース当日〜翌日の朝7〜8時（通勤・通学時間帯）
- 実録・ノウハウ記事: 火曜〜木曜の20〜21時（夜のリラックスタイム）
- 避けるべき: 金曜夜〜日曜（週末はnoteのアクティブユーザーが減る傾向）

**表示フォーマット:**
```
⏰ 投稿タイミング提案:
  推奨: 火曜 20:00〜21:00
  理由: 過去の同タイプ記事で火曜20時台のスキ数が最多
  （データ不足のためデフォルト推奨値を使用）
```

### 関連記事リンク

**選定基準:**
- テーマの類似性（同じ技術・ツールを扱っている）
- シリーズ性（前編/後編、入門/応用の関係）
- 既存記事のうちスキ数が多いものを優先

**挿入フォーマット（HTMLに含める）:**

```html
<li><a href="{noteURL}">{記事タイトル}</a></li>
```


### サムネイル生成プロンプト（Gemini ImageFX / Imagen 対応）

記事の内容に合わせたサムネイル画像生成用のプロンプトを出力する。
プロンプト設計の詳細リファレンスは `gemini_prompt_guide.md` を参照。

**プロンプト構造（5要素）:**
以下の順序で組み立てる。場面を描写する文章型で書き、キーワードの羅列にしない。

```
[被写体と特徴] + [背景・環境] + [スタイル] + [照明] + [構図・技術仕様]
```

**記事タイプ別スタイル選択:**

| 記事タイプ | 推奨スタイル | カラー方向 |
|---|---|---|
| 速報（sokuho） | `cinematic digital art` or `futuristic UI screenshot style` | ダーク背景 + ネオンアクセント（cyan, electric blue） |
| 実録（jituroku） | `editorial photography style` or `photorealistic` | ナチュラルトーン、温かみのあるライティング |
| ノウハウ（knowhow） | `modern flat illustration` or `isometric illustration` | クリーン、ブランドカラー（deep blue + cyan + orange） |

**ブランドカラー（デフォルト）:** deep navy blue + electric cyan + orange accents
記事の雰囲気に合わせてカラーパレットを変更してよいが、テック感・先進性は維持する。

**必須ルール:**
- 英語で出力（Gemini / Imagen は英語プロンプトの方が精度が高い）
- テキスト除外はセマンティック表現を使う: `clean image with no typography or labels`（「No text」だけより効果的）
- noteサムネイル用にネガティブスペースを確保: `negative space on the right third for title text` 等
- アスペクト比: **`16:9` 固定**（note推奨サイズ: 1280×670px）。`1:1` は使わない
- 人物が含まれる場合: `young university student`（大学生の設定に合わせる）

**照明の使い分け:**
- 速報記事: `neon lighting`, `dramatic lighting`, `rim lighting`
- 実録記事: `golden hour lighting`, `natural lighting`, `soft diffused light`
- ノウハウ記事: `studio lighting`, `soft diffused light`, `clean even lighting`

**構図の指定:**
- テキスト挿入用の余白を必ず確保する（`negative space on [位置] for text overlay`）
- 被写体は中央寄りまたは三分割法で配置
- `shallow depth of field`（背景ボケ）でメイン被写体を強調

**出力フォーマット:**
```
🎨 サムネイルプロンプト:
[英語プロンプト（場面描写型、3〜5文）]

📐 推奨設定: アスペクト比 16:9（note推奨 1280×670px） / 解像度 2K
🎯 生成先: Gemini ImageFX（https://aitestkitchen.withgoogle.com/tools/image-fx）
```

## Phase 9: PDCAフィードバック記録

`context/performance.md` に以下を追記するよう促す:
- 記事タイトル・URL・公開日
- 投稿後1週間のPV・スキ数
- 売上（有料記事の場合）

このデータは次回のPhase 2（トピック選定）で自動参照される。

### Notion記録: パフォーマンス更新の案内
記事投稿後、以下をNotionに記録するよう促す:
- noteURL: 投稿後のURL
- 公開日: 投稿日
- 投稿1週間後に以下を更新:
  - PV: ページビュー数
  - スキ数: スキの数
  - 売上: スキ数 × 価格 × 1.5
  - メモ: 学び・次回への示唆

### PDCAメモの記録
記事管理DBのメモ欄に、以下のフォーマットでPDCA記録を残すよう促す:

**投稿直後に記録:**
- 投稿日時
- noteURL
- 使ったタグ一覧

**1週間後に記録:**
- PV数
- スキ数
- コメント数
- 高評価数（購入手続きバナー下に表示）
- 売上推計（スキ数 × 価格 × 1.5）

**振り返り:**
- 何が良かったか（タイトル？内容？タイミング？）
- 何が悪かったか
- 次回への改善アクション（具体的に1つ）

このPDCAデータは次回の /note-run 実行時にPhase 0で自動参照される。

### voice-samples.md 自動更新

Phase 0で取得したNotionの記事管理DBから、**スキ数上位3記事**を特定し：
1. 該当記事の草稿ファイル（drafts/内）を読み込む
2. 導入部・まとめ部・印象的な一文を抽出
3. `context/voice-samples.md` の「サンプル文」セクションに追記する
4. 既に登録済みのサンプルと重複する場合はスキップ
5. サンプルが10個を超えたら、スキ数が低い記事のサンプルを削除（最新・高パフォーマンスを優先）

これにより、成功した記事の文体が自動的に学習される。
